## Kontext

Im Offline-Modus von GradeHub erhält eine Lehrkraft ihre ENM-JSON-Datei (`enm.json.gz`)
nicht über eine direkte Serververbindung, sondern als Datei — typischerweise per
schulischem E-Mail-System oder über einen Intranet-Dateiserver.

Diese Datei enthält **besonders schutzbedürftige personenbezogene Daten**
(Noten, Leistungsstand, persönliche Schülerdaten) im Sinne der DSGVO Art. 9 Abs. 1.
Der Transport über unverschlüsselte oder nur transportverschlüsselte Kanäle
(TLS endet am Mailserver — die Datei liegt dann im Klartext im Postfach) ist
datenschutzrechtlich problematisch.

### Rahmenbedingungen

| Bedingung | Ausprägung |
|-----------|------------|
| Exporteur | Schulverwaltung/Admin **oder** Lehrkraft selbst |
| Transportweg | Schulisches E-Mail-System, Intranet-Server |
| Verschlüsselungskontext | Ausschließlich im Browser — kein Setup, kein externes Tool für Lehrkräfte |
| Zielgruppe | Lehrkräfte ohne technische Vorkenntnisse |

### Das Problem in einem Satz

E-Mail-Transportverschlüsselung (TLS) schützt den **Übertragungsweg**, nicht die
**ruhenden Daten** im Postfach. Eine ENM-Datei im Anhang liegt im Mailserver-Postfach
im Klartext vor. Das ist für Schülerdaten nicht akzeptabel.

---

## Entscheidung

### Grundprinzip: Encrypt-before-send

Die ENM-Datei wird **vor dem Versand verschlüsselt** — direkt im Browser,
ohne externes Tool. Empfang und Entschlüsselung erfolgen ebenfalls im Browser,
beim Upload in GradeHub. Das Passwort wird **niemals** zusammen mit der Datei
übertragen, sondern auf einem separaten Kanal kommuniziert.

```
[SVWS-Export]
     │
     ▼
[Browser: Verschlüsselung]  ←── Passwort (separater Kanal: Telefon / Messenger)
     │
     ▼
[Datei: enm.json.gz.enc]
     │
     ├── per E-Mail
     └── per Intranet-Share
          │
          ▼
     [Lehrkraft-Browser: Upload + Entschlüsselung]  ←── Passwort eingeben
          │
          ▼
     [GradeHub: Noteneingabe / Zeugnisvorbereitung]
```

---

### Kryptografische Verfahren (Web Crypto API — nativ im Browser)

GradeHub nutzt ausschließlich die **Web Crypto API** (`window.crypto.subtle`),
die ohne npm-Pakete in jedem modernen Browser verfügbar ist.

#### Schlüsselableitung: PBKDF2

Aus dem vom Benutzer eingegebenen Passwort wird ein kryptografischer Schlüssel
abgeleitet. Ein zufälliges **Salt** wird mit der Datei gespeichert (kein Geheimnis,
schützt gegen Rainbow-Table-Angriffe).

```typescript
async function deriveKey(password: string, salt: Uint8Array): Promise<CryptoKey> {
  const enc = new TextEncoder()
  const keyMaterial = await crypto.subtle.importKey(
    'raw', enc.encode(password), 'PBKDF2', false, ['deriveKey']
  )
  return crypto.subtle.deriveKey(
    {
      name: 'PBKDF2',
      salt,
      iterations: 310_000,   // OWASP-Empfehlung 2024 für PBKDF2-SHA-256
      hash: 'SHA-256',
    },
    keyMaterial,
    { name: 'AES-GCM', length: 256 },
    false,
    ['encrypt', 'decrypt']
  )
}
```

#### Verschlüsselung: AES-256-GCM

AES-GCM bietet **Vertraulichkeit und Integrität** (authentifizierte Verschlüsselung).
Ein zufälliger **IV** (Initialization Vector) wird pro Verschlüsselung neu generiert
und mit der Datei gespeichert.

```typescript
async function encryptENM(plaintext: ArrayBuffer, password: string): Promise<ArrayBuffer> {
  const salt = crypto.getRandomValues(new Uint8Array(16))
  const iv   = crypto.getRandomValues(new Uint8Array(12))
  const key  = await deriveKey(password, salt)

  const ciphertext = await crypto.subtle.encrypt(
    { name: 'AES-GCM', iv },
    key,
    plaintext
  )

  // Dateiformat: [magic 8B][salt 16B][iv 12B][ciphertext]
  const result = new Uint8Array(8 + 16 + 12 + ciphertext.byteLength)
  result.set(new TextEncoder().encode('GRADEHUB'), 0)  // Magic Bytes
  result.set(salt, 8)
  result.set(iv, 24)
  result.set(new Uint8Array(ciphertext), 36)
  return result.buffer
}
```

#### Dateiformat der verschlüsselten Datei

| Offset | Länge | Inhalt |
|--------|-------|--------|
| 0      | 8 B   | Magic Bytes `GRADEHUB` (Erkennung + Versionsraum) |
| 8      | 16 B  | Salt (zufällig, Klartext) |
| 24     | 12 B  | IV / Nonce (zufällig, Klartext) |
| 36     | n B   | AES-256-GCM-Ciphertext (inkl. 16-Byte Auth-Tag) |

Dateiendung: `.enm.enc` — GradeHub erkennt diese Endung beim Upload und
zeigt automatisch das Passwort-Eingabefeld.

### Exportpfade

#### Pfad A: Schulverwaltung / Admin exportiert und verteilt

```
SVWS-Server
  └─► Admin-Browser: ENM abrufen → in GradeHub-Verschlüsselungs-Tool laden
        └─► Passwort festlegen (pro Lehrkraft oder global für den Jahrgang)
              └─► .enm.enc per schulischem E-Mail-System versenden
                    └─► Passwort per separatem Kanal (Telefon, Messenger) mitteilen
```

**Empfehlung:** Ein Passwort pro Lehrkraft (nicht ein globales Passwort),
damit der Verlust eines Passworts nicht alle Exportdateien kompromittiert.

#### Pfad B: Lehrkraft exportiert selbst

```
Lehrkraft meldet sich am SVWS-Webinterface an
  └─► Exportiert eigene ENM-Datei (enm.json.gz)
        └─► Öffnet GradeHub → verschlüsselt die Datei mit eigenem Passwort
              └─► Legt .enm.enc auf Intranet-Share oder speichert lokal
```

In diesem Pfad entfällt das Verteilungsproblem — die Lehrkraft verschlüsselt
für sich selbst und öffnet die Datei direkt in GradeHub.

---

### Passwort-Kanal-Trennung (Zwei-Kanal-Prinzip)

Das Passwort darf **niemals** zusammen mit der verschlüsselten Datei übertragen werden.

| Datei-Transport | Akzeptabler Passwort-Kanal |
|-----------------|---------------------------|
| E-Mail          | Telefonanruf, SMS, Schulkommunikationssystem (z. B. IServ-Nachricht) |
| Intranet-Share  | Mündlich, Hauspost, separate Nachricht im Intranet-Messenger |

GradeHub zeigt im UI einen entsprechenden **Hinweis** beim Passwort-Export:
> „Senden Sie das Passwort nicht zusammen mit der Datei. Nutzen Sie einen
> anderen Kommunikationsweg."

---

### Passwortanforderungen

GradeHub empfiehlt (und zeigt im Verschlüsselungs-Dialog):

- Mindestens 12 Zeichen
- Kombination aus Groß-/Kleinbuchstaben, Ziffern und Sonderzeichen
- Kein Schulname, Klassenkürzel oder Jahresangabe

Ein Passwort-Stärke-Indikator wird im Dialog angezeigt (rein clientseitig).

---

### Intranet-Server als sicherer Verteiler (Alternative zu E-Mail)

Wenn ein Intranet-Server (z. B. nginx, Apache, Nextcloud) verfügbar ist,
ist dieser Weg **vorzuziehen**, da der Zugriff durch Schulaccounts geschützt
werden kann:

```
Admin legt .enm.enc in passwortgeschütztem Intranet-Ordner ab
  └─► Lehrkraft greift mit ihrem Schulaccount auf den Ordner zu
        └─► Download der .enm.enc
              └─► Öffnen in GradeHub → Passwort eingeben → fertig
```

In diesem Szenario reicht ein gemeinsames Verschlüsselungspasswort für alle
Lehrkräfte einer Schule, da der Zugriff auf die Datei durch die Intranet-Authentifizierung
gesichert ist (Defense in Depth).

## Verworfene Alternativen

| Alternative | Grund für Ablehnung |
|-------------|---------------------|
| **PGP / S/MIME** | Erfordert Schlüsselverwaltung und Tool-Setup — nicht zumutbar für Lehrkräfte ohne technischen Hintergrund |
| **ZIP mit Passwort (AES)** | Würde ein externes Tool (7-Zip o. ä.) erfordern — widerspricht der Browser-Only-Anforderung aus ADR-0001 |
| **TLS-Transport als ausreichend betrachten** | TLS endet am Mailserver; Datei liegt im Postfach im Klartext — datenschutzrechtlich nicht akzeptabel |
| **Drittanbieter-Cloud (z. B. Google Drive)** | DSGVO-Problematik bei personenbezogenen Schülerdaten; nicht entschieden, da schulische Infrastruktur vorhanden |
| **npm-Kryptobibliothek (z. B. libsodium.js)** | Web Crypto API ist nativ, auditiert, kein zusätzliches Paket nötig |

## Implementierungshinweise

### GradeHub-UI: Verschlüsselungs-Dialog

Ein separates Modal (erreichbar über „Datei verschlüsseln" auf der Startseite)
bietet:

1. Datei auswählen (`.enm.json.gz`)
2. Passwort eingeben + Stärke-Indikator
3. Passwort wiederholen
4. Hinweis: „Passwort auf separatem Kanal übermitteln"
5. Button: „Datei verschlüsseln und herunterladen" → erzeugt `.enm.enc`

### GradeHub-UI: Upload-Dialog (Entschlüsselung)

Erkennt `.enm.enc` automatisch an der Dateiendung und dem Magic-Byte-Header:

1. Datei auswählen
2. Passwort-Feld erscheint automatisch
3. Entschlüsselung im Browser → ENM-Daten in den Store laden
4. Bei falschem Passwort: AES-GCM Auth-Tag schlägt fehl → klare Fehlermeldung

### Technische Einschränkungen

- Die Web Crypto API erfordert einen **Secure Context** (`https://` oder `localhost`).
  Bei `file://`-Betrieb (Offline-Build) ist `crypto.subtle` in den meisten Browsern
  **nicht verfügbar**. → Die Verschlüsselungsfunktion ist nur im Webserver-Deployment
  oder im `localhost`-Entwicklungsmodus nutzbar.
- **Konsequenz für den Build:** Der Offline-Build (`file://`) kann `.enm.enc`-Dateien
  **nicht entschlüsseln**. Er unterstützt ausschließlich den Upload unverschlüsselter
  `.enm.json.gz`-Dateien. GradeHub zeigt im `file://`-Modus einen entsprechenden
  Hinweis.

## Datenschutz-Bewertung

| Anforderung | Erfüllt durch |
|-------------|---------------|
| DSGVO Art. 5 Abs. 1 f (Integrität und Vertraulichkeit) | AES-256-GCM mit Authentifizierung |
| Kein Datenabfluss an Dritte | Vollständige Browser-Verarbeitung, kein externes API |
| Pseudonymisierung / Datensparsamkeit | ENM enthält nur die für die Lehrkraft relevanten Daten |
| Technisch-organisatorische Maßnahmen (TOMs) | Zwei-Kanal-Prinzip + PBKDF2-Schlüsselableitung |
