---

## Kontext

GradeHub speichert alle Änderungen einer Noteneingabe-Sitzung im lokalen
Change-Buffer (ADR-0001, E4). Am Ende einer Sitzung müssen diese Änderungen
dauerhaft in den SVWS-Server zurückgeschrieben werden.

Der SVWS-Server stellt dafür einen dedizierten Import-Endpunkt bereit:

```
POST /db/{schema}/enm/v2/import
```

Dieser Endpunkt nimmt ENM-Notendateien entgegen — **auch von einzelnen Lehrkräften**,
nicht nur vollständige Schulexporte. Das ist die architekturelle Grundlage für
GradeHub: jede Lehrkraft kann ihre eigenen Leistungsdaten eigenständig zurückschreiben,
ohne dass andere Lehrkräfte davon betroffen sind.

Das ADR beschreibt, wie GradeHub diesen Endpunkt nutzt, wie Timestamps die
Datenkonsistenz sichern und wie der Offline-Modus ohne direkte Serververbindung
abgewickelt wird.

---

## Entscheidung

### Grundprinzip: Timestamp-gesicherte ENM-Datei als Transportformat

GradeHub nutzt **das ENM-Format selbst** als Rückschreibe-Format — kein
proprietäres Delta-Format. Die Lehrkraft hat beim Export eine ENM-Datei erhalten
(oder per API geladen); GradeHub befüllt diese mit den eingegebenen Noten und
sendet sie über den Import-Endpunkt zurück. Der SVWS-Server kennt das Format
und übernimmt die Konsolidierung.

Jede Änderung im Change-Buffer wird mit einem **ISO-8601-Timestamp (UTC)**
versehen. Dieser dient als Grundlage für die serverseitige Konfliktauflösung
nach dem **Last-Write-Wins**-Prinzip: der Server bevorzugt Werte mit neuerem
Timestamp.

```typescript
// Change-Buffer-Eintrag im Pinia-Store
interface LeistungsChange {
  schuelerId:        number
  lerngruppenId:     number
  feld:              LeistungsFeld
  alterWert:         string | null   // Wert aus dem ENM-Export (Snapshot-Basis)
  neuerWert:         string | null   // Eingegebener Wert der Lehrkraft
  geaendertAm:       string          // ISO 8601 UTC: "2026-04-04T14:23:07Z"
  geaendertVon:      string          // Lehrkraft-Kürzel aus dem ENM
  enmBasisTimestamp: string          // Timestamp des ENM-Exports (Snapshot-Zeitpunkt)
}

type LeistungsFeld =
  | 'note'
  | 'noteQuartal'
  | 'fehlstundenGesamt'
  | 'fehlstundenUnentschuldigt'
  | 'fachbezogeneBemerkungen'
  | 'istGemahnt'
  | 'mahndatum'
```

---

### Modus A — Online-Rückschreiben (direkte Verbindung zum SVWS-Server)

Die Lehrkraft ist während der Sitzung mit dem SVWS-Server verbunden.
GradeHub führt am Sitzungsende den Import direkt durch.

#### Ablauf

```
Lehrkraft schließt Eingabe ab
  └─► GradeHub: Änderungszusammenfassung anzeigen
        └─► Lehrkraft bestätigt: "An SVWS übertragen"
              └─► GradeHub baut ENM-Rückschreibe-Datei auf
                    (Original-ENM + Change-Buffer angewendet, mit Timestamps)
                    └─► POST /db/{schema}/enm/v2/import
                          Authorization: Basic base64({username}:{password})
                          Content-Type: application/json
                                │
                                ├── 200 OK → Importprotokoll anzeigen
                                ├── 409 Conflict → Konflikte anzeigen
                                └── Netzwerkfehler → Fallback: Offline-Export anbieten
```

#### Sicherungskopie immer anbieten

Unabhängig vom Ergebnis des API-Calls bietet GradeHub **immer** den Download
der fertig befüllten ENM-Datei an. Diese dient als Nachweis und Sicherungskopie
der eingegebenen Daten für die Lehrkraft.

---

### Modus B — Offline-Rückschreiben (Datei-basierter Workflow)

Die Lehrkraft hat keine direkte Serververbindung. Die befüllte ENM-Datei
wird exportiert und über einen sicheren Kanal (ADR-0005) an die Schulverwaltung
zurückgegeben. Der Admin spielt die Datei über den Import-Endpunkt ein.

#### Ablauf Lehrkraft

```
Lehrkraft schließt Eingabe ab
  └─► GradeHub: Änderungszusammenfassung anzeigen
        └─► Lehrkraft klickt "Datei exportieren"
              └─► Download: enm_rueck_{kuerzel}_{datum}.json.gz
                    └─► Datei verschlüsseln (ADR-0005) → .enm.enc
                          └─► Sicher übermitteln (E-Mail / Intranet-Share)
```

#### Ablauf Admin / Schulverwaltung

```
Admin empfängt .enm.enc
  └─► Entschlüsseln → enm_rueck_{kuerzel}_{datum}.json.gz
        └─► POST /db/{schema}/enm/v2/import
              (im SVWS-Webclient oder per Admin-Skript)
              └─► Importprotokoll prüfen und archivieren
```

### Timestamp-Strategie und Konfliktauflösung

#### Wann entstehen Timestamps?

| Zeitpunkt | Feld |
|-----------|------|
| ENM-Export vom Server | `enmBasisTimestamp` (aus der ENM-Datei selbst) |
| Lehrkraft ändert einen Wert | `geaendertAm` (Moment der Eingabe, Browser-UTC) |
| GradeHub baut Rückschreibe-Datei | `exportTimestamp` (Moment des Exports) |

#### Konfliktszenarien beim serverseitigen Import

| Situation | Behandlung |
|-----------|------------|
| Feld nur von dieser Lehrkraft geändert | Konfliktfrei, Wert wird übernommen |
| Feld server-seitig geändert, `geaendertAm` ist **neuer** | Last-Write-Wins: Lehrkraft-Wert gewinnt |
| Feld server-seitig geändert, `geaendertAm` ist **älter** | Server-Wert bleibt, Protokolleintrag |
| `enmBasisTimestamp` liegt weit in der Vergangenheit | Warnung, manuelle Admin-Freigabe empfohlen |

GradeHub zeigt das vom Server zurückgegebene Importprotokoll im UI an.
Konflikte werden deutlich hervorgehoben mit der Möglichkeit, einzelne
Werte manuell zu überschreiben und erneut zu importieren.

### Aufbau der Rückschreibe-ENM-Datei

GradeHub wendet den Change-Buffer auf die im Pinia-Store gehaltene Original-ENM-Struktur
an und erzeugt eine valide ENM-Datei, die **ausschließlich die Leistungsdaten
der eigenen Lerngruppen** enthält — keine Daten anderer Lehrkräfte.

```typescript
async function buildRueckschreibeENM(
  originalEnm: ENMDaten,
  changeBuffer: Map<string, LeistungsChange>
): Promise<ENMDaten> {

  // Tiefe Kopie — Original-ENM im Store bleibt unberührt
  const rueckEnm = structuredClone(originalEnm)
  rueckEnm.exportTimestamp = new Date().toISOString()

  for (const leistung of rueckEnm.leistungsdaten) {
    const key = `${leistung.schuelerID}:${leistung.lerngruppenID}`
    const change = changeBuffer.get(key)
    if (!change) continue

    // Felder mit Timestamps übernehmen
    if (change.note !== undefined) {
      leistung.note = change.note
      leistung.noteTimestamp = change.geaendertAm
    }
    if (change.fehlstundenGesamt !== undefined) {
      leistung.fehlstundenGesamt = change.fehlstundenGesamt
      leistung.fehlstundenTimestamp = change.geaendertAm
    }
    // ... weitere Felder analog
  }

  return rueckEnm
}
```

Die fertige ENM-Struktur wird vor dem POST gzip-komprimiert (analoges Format
zum Export-Endpunkt `GET /enm/v2/alle/gzip`).

---

### Keine Echtzeit-Synchronisation

GradeHub überträgt Änderungen **nicht kontinuierlich** während der Sitzung.
Der Import erfolgt **ausschließlich am Sitzungsende**, explizit durch die
Lehrkraft ausgelöst — als atomare Transaktion.

**Begründung:**
- Halbfertige Noteneingaben gelangen nicht auf den Server
- Kein inkonsistenter Zwischenzustand (z. B. Note ohne Fehlstunden)
- Klares Transaktionsmodell: eine Sitzung = ein Import

#### UI-Schutz gegen versehentlichen Datenverlust

GradeHub zeigt während der gesamten Sitzung einen persistenten Statusindikator:

```
┌──────────────────────────────────────────────────────────────┐
│  ● 14 nicht übertragene Änderungen    [Exportieren / Senden] │
└──────────────────────────────────────────────────────────────┘
```

Beim Versuch, den Browser-Tab zu schließen oder zu navigieren,
erscheint ein `beforeunload`-Hinweis auf nicht gesicherte Änderungen.

---

### Authentifizierung am Import-Endpunkt

Der Import-Endpunkt verwendet **BasicAuth**, identisch zum Export-Endpunkt
(bereits in SVWS-Conference implementiert und erprobt):

```
Authorization: Basic base64({username}:{password})
```

Die Zugangsdaten werden ausschließlich im Pinia-Store der laufenden Browser-Session
gehalten — kein `localStorage`, kein `sessionStorage`, kein Cookie.
Sie werden **nicht** in der exportierten ENM-Datei gespeichert.

Im Offline-Modus führt der **Admin** den Import mit seinen eigenen,
höher privilegierten Zugangsdaten durch.

---

### Versionserkennung und Stabilität des Endpunkts

GradeHub nutzt explizit `/enm/v2/import`. Sollte der SVWS-Server zukünftig
eine `v3`-Version einführen, prüft GradeHub beim Verbindungsaufbau, welche
Version verfügbar ist, und wählt die höchste:

```typescript
async function detectENMVersion(baseUrl: string, schema: string): Promise<'v2' | 'v1'> {
  // HEAD-Request: 401 = Endpunkt existiert aber Auth fehlt → Version vorhanden
  const v2 = await fetch(`${baseUrl}/db/${schema}/enm/v2/alle/gzip`, { method: 'HEAD' })
  if (v2.ok || v2.status === 401) return 'v2'
  return 'v1'
}
```

## Verworfene Alternativen

| Alternative | Grund für Ablehnung |
|-------------|---------------------|
| **Separates Delta-JSON-Format** | Unnötig — der SVWS-Server akzeptiert ENM-Dateien nativ; ein eigenes Format würde eine zusätzliche serverseitige Verarbeitung erfordern |
| **PATCH auf einzelne Leistungsdatensätze** | Kein stabiler, für Drittanbieter freigegebener Endpunkt auf Einzelfeld-Ebene vorhanden |
| **Echtzeit-Synchronisation pro Tastendruck** | Halbzustände auf dem Server; SVWS bietet kein CRDT/OT-Modell |
| **Vollständige ENM-Datei aller Lehrkräfte zurücksenden** | Datenschutzproblem: Lehrkraft A hat keinen Zugriff auf Daten von Lehrkraft B; nicht nötig, da `/enm/v2/import` Einzellehrkraft-Dateien akzeptiert |
| **localStorage als Sicherungspuffer** | Nicht verfügbar im `file://`-Modus; nicht gewünscht (ADR-0001) |

---

## Risiken

| Risiko | Wahrscheinlichkeit | Maßnahme |
|--------|--------------------|----------|
| SVWS-Server ändert ENM-v2-Import-Semantik | Niedrig | Versionserkennung; E2E-Tests gegen `tests-server-enm` |
| Lehrkraft schließt Browser vor dem Import | Mittel | `beforeunload`-Dialog, persistenter Statusindikator |
| Konflikte bei paralleler Bearbeitung einer Lerngruppe | Niedrig | Last-Write-Wins mit Protokoll; Admin sieht Konfliktliste |
| Admin verliert Offline-Rückschreibe-Datei | Niedrig | Lehrkraft hält lokale Kopie; Dokumentation betonen |

---

## Implementierungsreihenfolge

1. **Change-Buffer mit Timestamps** im Pinia-Store (parallel zu ADR-0001, E4)
2. **`buildRueckschreibeENM()`** — ENM zusammenführen, gzip-komprimieren
3. **Import-Service** — POST gegen `/db/{schema}/enm/v2/import` mit BasicAuth
4. **Protokoll-Anzeige** nach Import (Erfolg, Konflikte, Fehler)
5. **`beforeunload`-Schutz** und persistenter Änderungs-Indikator im UI
6. **Offline-Export-Dialog** — Download + optionale Verschlüsselung (ADR-0005)
7. **Versionserkennung** beim Verbindungsaufbau
