## Kontext

SVWS-GradeHub ist ein neues Open-Source-Werkzeug für Lehrkräfte an deutschen Schulen. Es soll die **Noteneingabe und Zeugnisvorbereitung** direkt im Browser unterstützen — für Lerngruppen und Klassen aller Schulformen, mit einem expliziten Ausblick auf Grundschulzeugnisse (Textzeugnisse).

Das Projekt entsteht als Schwester-Applikation zu **SVWS-Conference** (Notenkonferenz-Übersicht), das bereits produktiv eingesetzt wird und als technische Referenz dient. Der Kern von SVWS-Conference: eine Vue-3-/TypeScript-/Vite-/Pinia-SPA, die ENM-Daten vom SVWS-Server oder per Datei-Upload bezieht und vollständig im Browser läuft — ohne eigenen Backend-Server.

GradeHub erweitert diesen Ansatz um einen **aktiven Schreibmodus**: Lehrkräfte sollen Noten eingeben, ändern und für die Zeugnisvorbereitung aufbereiten können, nicht nur lesen.

---

## Entscheidungen

### E1 — Frontend-Stack: Vue 3 + TypeScript + Vite + Pinia

**Entscheidung:** GradeHub übernimmt denselben Kern-Stack wie SVWS-Conference:

- **Vue 3** (Composition API) als UI-Framework
- **TypeScript** für vollständige statische Typisierung
- **Vite** als Build-Tool und Entwicklungsserver
- **Pinia** als zentrales State-Management

**Begründung:**

- Maximale Code-Wiederverwendung zwischen beiden Projekten (ENM-Typen, SVWS-API-Client, Utilities)
- Erprobte Kombination: SVWS-Conference läuft stabil auf diesem Stack seit mehreren Releases
- TypeScript ist für das SVWS-ENM-Datenmodell unverzichtbar — die Typdefinitionen (`enm.ts`) sind bereits vorhanden und können gegebenenfalls direkt übernommen werden
- Pinia ermöglicht eine saubere Trennung zwischen Originaldaten (readonly ENM-Store) und dem lokalen Änderungs-Buffer (Change-Store) — ein Muster, das sich in der Konferenz-App bewährt hat

**Verworfene Alternativen:**

- React/Next.js: kein Mehrwert, Code-Sharing mit SVWS-Conference würde entfallen
- Svelte: kleinere Ökosystem-Basis im schulischen OSS-Umfeld, Umstiegsaufwand nicht gerechtfertigt

---

### E2 — Browser-First-Architektur (kein eigener Applikationsserver)

**Entscheidung:** GradeHub läuft vollständig im Browser. Es gibt keinen eigenen Server-Prozess. Der Build-Output (statische Dateien) kann:

1. direkt per `file://` geöffnet werden (Offline-Modus)
2. auf jedem Webserver oder CDN bereitgestellt werden (GitHub Pages, nginx, Apache)

**Begründung:**

- Datenschutz: Schülerdaten verlassen den Browser nicht — kein Backend bedeutet keine neue Datenschnittstelle
- Infrastruktur: Schulen benötigen keine zusätzliche Server-Infrastruktur
- Deployment-Einfachheit: Update = neue statische Dateien einspielen
- Offline-Fähigkeit: Der bewährte Offline-Build-Ansatz aus SVWS-Conference (`scripts/build-offline.mjs`) wird übernommen

**Konsequenzen:**

- Kein serverseitiges Session-Management → State lebt nur in der Browser-Session
- Keine persistente Datenhaltung im Backend → Änderungen müssen am Ende der Sitzung exportiert oder zurück an den SVWS-Server übertragen werden
- CORS-Konfiguration am SVWS-Server bleibt Voraussetzung für den Online-Modus (wie in SVWS-Conference)

---

### E3 — Datenbezug: Dual-Modus (SVWS-REST-API oder Datei-Upload)

**Entscheidung:** GradeHub unterstützt zwei gleichwertige Eingabekanäle:

**Online-Modus:** REST-Abruf vom SVWS-Server via BasicAuth gegen die bekannten ENM-Endpunkte:
```
GET {SVWS-Basis-URL}/db/{schema}/enm/v1/lehrer/{id}
GET {SVWS-Basis-URL}/db/{schema}/enm/v2/lehrer/{id}
```

**Offline-Modus:** Upload einer `enm.teacher.json`-Datei (JSON), die die Lehrkraft vorab aus dem SVWS exportiert hat. Hier muss später eine Verschlüsselung bedacht werden.

**Begründung:**

- Identische Strategie wie SVWS-Conference — der Code für ENM-Abruf und -Parsing (`fflate` für gzip-Dekomprimierung) ist wiederverwendbar
- Lehrkräfte an Schulen ohne stabilen Server-Zugang (z. B. Heimarbeit) müssen trotzdem arbeitsfähig sein
- Keine neue API-Abhängigkeit: GradeHub nutzt ausschließlich den bereits durch SVWS-Conference verifizierten ENM-Endpunkt

**Erweiterungsvorbehalt:** Für das Zurückschreiben geänderter Noten an den SVWS-Server wird ein separater Write-Endpunkt benötigt. Dieser ist in ADR-0002 (Change-Export und Rückschreibe-Strategie) zu beschreiben, sobald die SVWS-API dafür stabil ist.

---

### E4 — Datenmodell und Schreibstrategie: Lokaler Change-Buffer

**Entscheidung:** GradeHub mutiert niemals die aus dem ENM gelesenen Originaldaten. Alle Änderungen (Noten, Zeugnisinhalte) werden in einem **separaten lokalen Change-Buffer** gehalten:

```typescript
// Schlüssel: `${schuelerId}:${lerngruppenId}`
type ChangeKey = `${number}:${number}`
type ChangeBuffer = Map<ChangeKey, Partial<LeistungsdatenPatch>>
```

Der Buffer lebt im Pinia-Store und wird am Sitzungsende als JSON exportiert oder (bei verfügbarer API) an den SVWS-Server übertragen.

**Begründung:**

- Bewährtes Muster aus SVWS-Conference
- Trennung von Lesen und Schreiben verhindert versehentliche Datenverluste
- Undo/Redo und "Was hat sich geändert?"-Ansicht sind damit trivial implementierbar
- Datenschutz: der Buffer enthält nur die Deltas, nicht den gesamten ENM-Dump

---

### E5 — Erweiterbarkeit: Grundschulzeugnisse (Textzeugnisse)

**Entscheidung:** Die Architektur wird von Beginn an modular angelegt, sodass ein zukünftiger Modus „Grundschulzeugnis" (freie Textzeugnisse ohne Ziffernoten) integriert werden kann, ohne den bestehenden Code umzubauen.

**Konsequenzen für die initiale Implementierung:**

- Der Zeugnis-/Notentyp wird durch ein `ZeugnisTyp`-Enum abstrahiert (`ZIFFERNOTEN | TEXTZEUGNIS`)
- Views und Stores werden gegen dieses Enum parametrisiert, nicht hart gegen Ziffernoten verdrahtet
- Die Grundschul-Implementierung erfolgt in einem späteren Release-Zyklus (eigenes ADR)

---

### E6 — Qualitätssicherung: TypeScript-Strict + ESLint + Vitest

**Entscheidung:** GradeHub übernimmt die QS-Werkzeuge aus SVWS-Conference:

| Werkzeug   | Zweck                              |
|------------|------------------------------------|
| `tsc --noEmit` | Statische Typprüfung           |
| ESLint     | Code-Qualität und Konsistenz       |
| Vitest     | Unit-Tests für Stores und Utilities|

TypeScript wird im Strict-Modus betrieben (`"strict": true` in `tsconfig.json`).

## Konsequenzen

**Positive Konsequenzen:**

- Code-Sharing mit SVWS-Conference von Tag 1 (ENM-Typen, API-Client, gzip-Parsing)
- Kein neuer Infrastrukturaufwand für Schulen
- Datenschutzkonformer Betrieb durch Browser-Only-Architektur
- Erweiterung auf Grundschulzeugnisse ist architekturell vorbereitet

**Negative Konsequenzen / Risiken:**

- Kein eigener Backend-Server bedeutet: persistente Speicherung nur über Export → Lehrkraft trägt Verantwortung für regelmäßigen Export
- Rückschreiben an SVWS hängt von der Verfügbarkeit eines stabilen Write-Endpunkts ab (externer Constraint)
- CORS-Konfiguration am Schul-Server bleibt eine Einstiegshürde für den Online-Modus
